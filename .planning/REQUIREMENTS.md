# Requirements: SPORE Self-Build Milestone

**Defined:** 2026-03-09
**Core Value:** SPORE must let an operator guide safe self-improvement with durable traceability, explicit review gates, and enough product surface to make the loop usable end to end.

## v1 Requirements

### Dashboard

- [x] **DASH-01**: Operator can open a dedicated self-build dashboard separate from general run-center views
- [x] **DASH-02**: Operator can see active work-item groups, blocked work items, pending proposals, pending validation, and recent follow-up signals from one self-build summary surface ✅ Plan 01
- [x] **DASH-03**: Operator can drill from self-build summary cards into linked goal plans, work-item groups, work items, work-item runs, and proposal artifacts ✅ Plan 01
- [x] **DASH-04**: Web and TUI self-build views consume the same orchestrator-composed read models rather than direct database or filesystem reads ✅ Plan 01

### Group Execution

- [x] **GRP-01**: Operator can define explicit dependencies between work items inside a work-item group
- [x] **GRP-02**: System blocks downstream work items until required dependencies complete successfully
- [x] **GRP-03**: System derives group-level readiness, blocked, running, failed, and completed state from child dependency state

### Lifecycle

- [ ] **FLOW-01**: Operator can materialize a goal plan into linked work-item groups and work items with durable lineage
- [ ] **FLOW-02**: Operator can start managed self-build work from the self-build flow without stitching separate commands or screens manually
- [x] **FLOW-03**: System preserves durable links across goal plans, work-item groups, work items, work-item runs, proposal artifacts, validation evidence, doc suggestions, and learning records ✅ Plan 01
- [ ] **FLOW-04**: Operator can follow one end-to-end supervised self-build lifecycle from goal creation through reviewed outcome

### Proposal Review

- [ ] **PROP-01**: Code-oriented work-item runs produce normalized proposal artifacts with summary, affected scope, linked files or docs, and validation status
- [ ] **PROP-02**: Reviewer can record proposal review notes and decision history on the proposal artifact
- [ ] **PROP-03**: Rejected proposals create explicit rework or retry linkage back to the originating work item and run
- [ ] **PROP-04**: Proposal review and approval states remain distinct from generic blocked or failed execution states

### Validation

- [ ] **VALD-01**: Work-item templates and project defaults can attach named validation bundles to self-build work before execution
- [ ] **VALD-02**: Validation bundles can reference existing scenario and regression definitions instead of ad hoc commands only
- [ ] **VALD-03**: Proposal review surfaces show required validation evidence, completed validation evidence, and remaining validation gaps
- [ ] **VALD-04**: System blocks proposal completion or approval when required validation evidence is missing or incomplete

### Safety And Defaults

- [ ] **SAFE-01**: Safe mode enforces allowed mutation scope server-side during planning, execution, and review
- [ ] **SAFE-02**: Operator can see whether self-build work is in allowed, restricted, or forbidden scope before approval
- [ ] **SAFE-03**: The SPORE project profile can define default governance, validation, and safe-mode behavior for self-build work

### Templates And Planning

- [ ] **TEMP-01**: Operator can bootstrap common SPORE self-build tasks from reusable work-item templates for docs, config or schema, operator surface, runtime validation, and review-only work
- [ ] **PLAN-01**: Goal planning uses project and domain context to recommend templates, validation bundles, and governance level conservatively

### Queue And Follow-Up

- [ ] **QUEUE-01**: Operator can view self-build queues for pending, blocked, running, waiting review, waiting approval, and failed work
- [ ] **QUEUE-02**: Queue entries expose next action, blocked reason, and linked lineage context
- [ ] **QUEUE-03**: Operator can filter or group queue views by goal plan, work-item group, or status
- [ ] **DOCS-01**: Operator can review documentation suggestions generated by self-build runs from a visible queue or drilldown surface
- [ ] **LEARN-01**: Operator can review learning records linked to work-item outcomes from the self-build surface
- [ ] **FAIL-01**: Operator can see repeated failure patterns across blocked work, validation failures, proposal rejections, and governance stalls

### Self-Build Validation Scenarios

- [ ] **SCEN-01**: Named supervised self-build scenarios exist for docs or config-only work, operator-surface work, validation-harness maintenance, and proposal-review flow validation

## v2 Requirements

### Expanded Autonomy

- **AUTO-01**: System can reprioritize queued self-build work automatically based on outcome history and dependency state
- **AUTO-02**: System can schedule speculative self-build work without direct operator initiation

### Deeper Self-Mutation

- **CORE-01**: Self-build can modify runtime-core orchestration internals outside the early safe-mode default scope

### Alternative Interfaces

- **CHAT-01**: Operator can drive self-build through a generic chat-first assistant interface in addition to structured work-item flows

## Out of Scope

| Feature | Reason |
|---------|--------|
| Unreviewed code mutation | Violates the supervised governance model that makes self-build safe enough to ship now |
| Direct client reads from SQLite or local artifact files | Breaks the existing thin-client HTTP boundary and duplicates server contract logic |
| Broad runtime-core mutation in the first self-build milestone | Safe-mode enforcement and review depth need to mature before deeper mutation scope is allowed |
| Autonomous scheduling before dependency and queue semantics are stable | Adds complexity before the core supervised loop is trustworthy |
| Chat-only self-build UX as the main product surface | Would weaken durable linkage, review, and evidence compared with structured self-build flows |
