# Session Model

## Requirement

Every active agent runs in a live session that is inspectable by operators.

## Core Concepts

- Agent identity: stable logical agent definition.
- Agent session: concrete runtime instance tied to task/workflow context.
- Session lane: communication and control channel per live execution.

## Lifecycle

1. planned
2. starting
3. active
4. paused
5. completed | failed | canceled
6. archived

## Metadata

Each session links to:
- project,
- domain,
- team,
- workflow template,
- task/goal reference,
- role profile,
- parent/child relationship.

Workflow and coordination metadata may also associate the session with:

- an execution id,
- a parent execution id,
- a coordination group id when the session belongs to grouped work.

## Inspectability Targets

- tmux-backed execution for terminal inspection,
- session transcript capture,
- status and event feed,
- web visibility through gateway and browser operator surfaces.

## Current Executable Slice

The repository now includes:

- live session plans generated by `packages/runtime-pi/`,
- tmux-backed runtime launches,
- session metadata persisted by `packages/session-manager/`,
- lifecycle transitions and event emission to a local NDJSON event log,
- PI JSON event capture and PI session artifacts in `tmp/sessions/`,
- PI RPC status and control queue artifacts in `tmp/sessions/`,
- detached-session reconciliation from exit files,
- operator inspection via `packages/tui/`,
- session, artifact, and event-stream access plus narrow control actions via `services/session-gateway/`.

Workflow execution now also links parent and child sessions so a single execution can be rendered as a step/session tree in the web surface.

That session tree should not be confused with the durable workflow-execution graph:

- sessions explain runtime activity,
- executions explain workflow state and lineage,
- coordination groups explain which executions are being managed together.

Workflow-level `paused` and `held` states should therefore be interpreted from orchestrator execution state, not inferred from session status alone. A grouped workflow may be intentionally held even when no live runtime session is currently active.

This is still not the final session broker, but it is a real executable session loop rather than a documentation-only placeholder.
